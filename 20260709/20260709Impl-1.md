# 唐诗地图（长安诗境）— 全量实施执行计划

> 本文档为 `/goal` 命令的目标说明书（goal spec）：定义了目标、输入材料、任务清单执行方式、单张任务的标准作业流程、验收测试方法、以及终止/升级条件。执行者（agent）应按本文档自主推进，直至"终止条件"全部满足或触发"升级条件"。

## 0. 目标（Goal）

将已评审定稿的《PRD-唐诗地图可视化项目.md》《Design-唐诗地图可视化项目.md》（v1.1）落地为一个**可公网访问、满足 PRD 第 10 节全部验收标准**的唐诗地图静态站点，并在过程中保持 `tickets.md` 任务状态与代码、数据、文档同步一致。

## 1. 输入材料（执行前必读，禁止跳过）

| 文件/目录 | 内容 | 用途 |
| --- | --- | --- |
| `20260709/PRD-唐诗地图可视化项目.md` | 产品需求文档（功能、数据、验收标准） | 理解"做什么"与"验收标准" |
| `20260709/Design-唐诗地图可视化项目.md` | 技术设计文档（架构、模块划分、技术选型） | 理解"怎么做" |
| `tickets.md` | 拆分好的任务清单（T1–T13，含依赖关系与勾选项） | 执行主线，任务状态的唯一权威来源 |
| `CONTEXT.md` | 项目术语表（ubiquitous language） | 命名、文案、测试用例须与术语表一致，禁止使用"Avoid"标注的别名 |
| `docs/adr/0001–0008` | 已裁决的架构决策（四段式时期、地点中心标记、WGS84/GCJ02、多地点关联、事件总线、核心/扩展分层、数据生产策略、配图流水线） | 实现细节的强制约束，如与 PRD/Design 正文冲突以 ADR 为准 |

执行第一步：完整阅读上述五类材料，在心中（或简短记录中）确认对需求与设计已形成清晰、一致的理解，再进入任务执行阶段。**不要求产出额外的理解文档**，除非发现材料之间存在矛盾——此时应先记录矛盾点，按"终止/升级条件"处理，而非自行臆断取舍。

## 2. 任务主线：按 `tickets.md` 顺序执行 T1–T13

`tickets.md` 中共 13 张 ticket（T1–T13），每张都是一条可独立验证的 tracer-bullet 垂直切片。执行规则：

1. **依赖顺序**：严格按每张 ticket 的 `Blocked by` 字段推进，禁止在阻塞票未完成前开工后置票。默认执行顺序为 T1 → T2 → T3 → T4 → T5 → T6 → T7 → T8 → T9 → T10 → T11 → T12 → T13；若执行环境支持并行子任务，T1 完成后 **T2（代码主线）与 T10（数据主线）可并行**，但若只能单线程推进，则按编号顺序依次完成，不得因追求并行而打乱依赖关系。
2. **一张一清**：一张 ticket 完成并通过验收后，才开始下一张；每张 ticket 完成后清空/重置无关上下文，避免上下文污染影响后续判断。
3. **禁止合并 ticket**：不得将多张 ticket 的实现混在一次改动中提交；每张 ticket 对应清晰、可回溯的一次（或若干次）提交。
4. **禁止扩大范围**：每张 ticket 的实现范围以其"What to build"与勾选项为界，不得附带实现属于其他 ticket 的功能（scope creep），也不得遗漏勾选项中的任意一条。

## 3. 单张 ticket 的标准作业流程（SOP）

对 **每一张** ticket，必须按以下顺序执行，不得省略或调换步骤：

1. **理解范围**：重读该 ticket 的 "What to build" 与勾选项，定位其在 PRD/Design 中对应的功能编号（如 F1–F12）或数据/非功能需求条款，以及相关 ADR。
2. **调用 `/implement` 技能**：以该 ticket 的勾选项作为实现规格，驱动实现。`/implement` 会在预先约定的接口边界（seam）处应用 **`/tdd`**（红→绿，一次一个切片，先写失败测试再写最小实现），并在编码完成后自动触发 **`/code-review`**（Standards 轴 + Spec 轴双维度审查，对照本 ticket 范围与项目既有编码规范）。因此执行顺序固定为：`/implement` → （内含）`/tdd` 逐切片红绿 → （内含）`/code-review` → 提交。**不得跳过 `/tdd` 直接手写实现，也不得跳过 `/code-review` 直接提交。**
3. **静态校验**：涉及数据改动的 ticket，运行 `scripts/validate-data.mjs`，确认退出码为 0；涉及代码改动的 ticket，本地打开/运行站点做一次手工 smoke check，确认没有明显的控制台报错或空白页。
4. **自查勾选项**：逐条比对该 ticket 的勾选项，确认全部已实现且已被测试或校验覆盖，未完成的勾选项禁止标记为完成。
5. **同步任务状态**（见第 4 节）：在 `tickets.md` 中勾选该 ticket 已完成的子项。
6. **提交代码**：`/code-review` 通过后提交到当前分支，提交信息清晰说明本次完成的是哪张 ticket、做了什么。

## 4. 任务状态同步规则

- `tickets.md` 是任务状态的**唯一权威来源**。执行过程中每完成一项勾选项，立即在 `tickets.md` 对应行将 `- [ ]` 改为 `- [x]`，不要批量延后更新。
- 一张 ticket 的全部勾选项都变为 `- [x]` 后，视为该 ticket 完成，方可开始下一张。
- 如某张 ticket 因故被拆分、调整范围或延期，须在 `tickets.md` 中该 ticket 标题下用简短说明记录原因，而不是silent地跳过。
- 严禁在未真正实现/验证的情况下预先勾选（禁止"假勾选"）。

## 5. 数据主线特别说明（T10–T12）

T10–T12 是数据生产与配图流水线，与 T2–T9 的前端功能开发同属主线但性质不同：

- T10、T11 的"实现"指按 `docs/adr/0007-数据生产策略ai起草人工校验深浅分级.md` 的分级方法产出/扩充 `data/` 下的 JSON 数据（诗人、地点、诗词），而非编写业务代码；仍需对每次数据批次运行 `scripts/validate-data.mjs` 并确保退出码 0，可视为该场景下的"测试"，替代常规单元测试；`/code-review` 的 Spec 轴对照 PRD §4/§6 的数据字段与规模要求逐条核对。
- T12 的"配图"环节使用 Grok 生图模型按 `docs/adr/0008-配图占位与grok生图流水线.md` 约定的统一唐风 art-direction 生成配图，完成后需要将对应地点/诗词记录中的 `images[].status` 由占位状态更新为 `generated`。
- 抽查一致性时应覆盖 tickets.md 中列出的具体样例（李白/杜甫→盛唐；马嵬→extended；《长恨歌》名句数组≥3 且仙界不建标记；《送元二》渭城 core + 阳关 related）。

## 6. 全部 ticket 完成后：端到端验收测试

T1–T13 全部完成并逐条勾选后，进入验收阶段，目标是逐条核验 PRD 第 10 节"验收标准"：

1. **功能与联动**：使用浏览器自动化工具（如可用的浏览器自动化/Browser Use 能力）打开部署或本地构建的站点，依次验证：
   - 地图正常加载、可缩放平移、"回到长安"复位可用；
   - F1–F5（地图、标记、详情面板、时期/诗人筛选、列表侧栏）全部可用且互相联动正确；
   - 「扩展范围」开关能正确显隐远地标记并调整 `maxBounds`；
   - F6 搜索、F7 时间轴、F8 诗人页、F9 古今对照图层（P1）均可用；
   - F10–F12（主题切换、深链分享、引导页，P2）均可用。
2. **数据规模与质量**：确认收录诗词 ≥ 60 首、地点 ≥ 30 处，`scripts/validate-data.mjs` 退出码为 0；随机抽检 ≥ 20% 数据条目核对诗文、作者、坐标无误。
3. **多端适配**：分别在桌面宽度与 375px 移动宽度下，完整走通"浏览 → 筛选 → 点击标记 → 阅读详情"核心路径，确认无布局错乱或功能缺失。
4. **性能与可访问性**：运行 Lighthouse，确认 Performance ≥ 80、Accessibility ≥ 90；如未达标，定位主要瓶颈（如图片未懒加载/未转 WebP、JSON 一次性全量加载等）并回到对应 ticket 补充修复。
5. **部署验证**：确认站点已部署到 Cloudflare Pages（或备选 GitHub Pages），公网地址可正常访问，`_headers`/`_redirects`/`404.html`/`sitemap.xml` 按 T13 要求就位。

验收过程中发现的任何不符合项，均视为对应 ticket 未真正完成：应回退该 ticket 状态（`tickets.md` 中取消勾选相关项并简述原因），重新走第 3 节 SOP 修复，而不是在验收文档里单独记账。

## 7. 终止条件（全部满足才算完成）

1. `tickets.md` 中 T1–T13 的**全部**勾选项均为 `- [x]`，且每张 ticket 的实现均已通过对应 `/code-review`；
2. 第 6 节验收测试的 5 项全部通过，PRD 第 10 节验收标准逐条达标；
3. 全部改动已提交到当前分支，`tickets.md` 状态与实际代码/数据状态一致，无遗留的"假勾选"或未记录的范围调整。

## 8. 升级条件（触发后立即停止并等待人工确认）

1. **同一问题连续尝试 3 次仍未解决**（如某项测试反复失败、某个环境/依赖问题反复出现、某条数据无法通过校验器等）：立即停止对该问题的继续尝试，不得进入第 4 次尝试或绕过问题静默跳过。
2. 触发升级时必须清晰说明：
   - 出现问题的 ticket 编号与具体勾选项；
   - 已尝试的 3 次方案与各自失败原因/证据（报错信息、校验器输出等）；
   - 该问题对后续 ticket（尤其是被其阻塞的 ticket）的影响范围；
   - 建议的可选方向（如需要人工提供数据来源、需要调整 ADR、需要放宽某验收指标等），但不擅自变更 PRD/Design/ADR 的既定决策。
3. 除上述"3 次未解决"外，若在材料阅读阶段发现 PRD/Design/tickets.md/CONTEXT.md/ADR 之间存在**实质性矛盾**（而非措辞差异），同样应在开工前停止并说明矛盾点，等待人工确认，不得自行裁定。

## 9. 全局约束

- 所有代码、数据字段命名、UI 文案、测试用例命名须与 `CONTEXT.md` 术语表一致，避免使用其"Avoid"栏标注的别名。
- 不在未产生新 ADR 的情况下擅自变更已裁决的架构决策（`docs/adr/0001–0008`）；如实现中发现某项 ADR 已不适用，按第 8 节升级条件处理，而非直接改写代码绕过。
- 每张 ticket 的实现须保持"数据与代码分离"的设计原则（Design §1.1）：新增诗词/地点只需追加 JSON，不应引入需要改代码才能生效的数据。
- 全程使用中文进行任务说明、提交信息与状态记录，与项目现有文档语言一致。
